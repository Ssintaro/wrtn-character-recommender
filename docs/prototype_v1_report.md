# 1차 프로토타입(v1) 구현 결과 및 한계점 진단

> 본 장은 뤼튼 크랙(Wrtn Character) 지능형 캐릭터 추천 시스템의 1차 프로토타입(v1) 구현 결과를 기술한다. RAG 파이프라인의 핵심 골격 수립과 50개 미러링 데이터셋 기반 모킹(Mocking)을 완료한 시점을 v1 마일스톤으로 정의하고, 실제 구동 로그를 근거로 시스템의 동작을 검증하는 한편, 공학적 관점에서 발견된 구조적 결함을 정량적으로 진단한다.

## 0. v1 마일스톤의 범위 정의 및 의도

본 프로토타입의 결과를 보고하기에 앞서, v1 마일스톤이 의도적으로 한정한 구현 범위(Scope)를 명시한다. 이는 후속 평가가 프로젝트 계획서의 최종 목표 전체가 아니라 v1이 책임지는 부분 목표를 기준으로 이루어지도록 하기 위함이다.

프로젝트 계획서가 정의하는 최종 시스템은 크게 두 개의 축으로 구성된다. 하나는 **(가) 사용자 발화 맥락을 분석하여 최적 캐릭터를 선별하는 지능형 매칭 엔진**이며, 다른 하나는 **(나) 매일 갱신되는 UGC 데이터를 수집·벡터화·반영하는 자동화 데이터 파이프라인(Scaffolding)** 이다. v1 마일스톤은 이 중 **(가) 매칭 엔진의 아키텍처 골격 수립에 범위를 집중**하며, (나) 자동화 파이프라인은 후속 마일스톤의 과제로 의도적으로 분리한다.

이러한 범위 분리는 다음의 판단에 근거한다.

- **검증 대상의 우선순위**: 추천 시스템의 본질적 가치는 매칭 로직의 타당성에 있다. 데이터 공급 경로가 자동화되기 이전에, 매칭 엔진이 구조적으로 올바르게 동작하는지를 먼저 검증하는 것이 공학적으로 합리적인 순서다.
- **데이터 수집의 독립적 과제 성격**: 실서비스 크롤링 및 실시간 동기화는 대상 플랫폼의 이용약관(ToS) 준수 검토를 선행 조건으로 요구하는 독립 과제다. 이를 매칭 엔진 검증과 분리함으로써, 각 과제를 고유한 제약 조건 위에서 다룰 수 있다.
- **인터페이스 안정성의 확보**: v1은 데이터가 적재되는 그릇(ChromaDB 메타데이터 스키마)과 매칭 파이프라인을 먼저 안정화한다. 이로써 후속 마일스톤에서 자동화 파이프라인이 연동될 때, 매칭 엔진 측 코드를 수정하지 않고 데이터 공급 계층만 교체할 수 있다.

따라서 v1은 **"수동 실행 기반의 매칭 엔진 프로토타입"** 으로 규정된다. 데이터 적재는 일회성 배치 스크립트(`build_index.py`)의 수동 호출로 수행되며, 자동 스케줄링·증분 갱신·실시간 동기화 등 계획서가 (나) 축에서 요구하는 기능은 v1 범위에 포함되지 않는다. 이 부분에 대한 구체적 추진 계획은 본 장 5절의 개선 로드맵에서 별도로 다룬다.

## 1. 1차 프로토타입 아키텍처 개요

v1 프로토타입은 사용자의 자연어 발화로부터 최적의 AI 캐릭터를 매칭하기 위해, **정적 프로필 제약(Static Profile Constraint)** 과 **동적 세션 파싱(Dynamic Session Parsing)** 을 결합한 5단계 하이브리드 필터링 파이프라인으로 설계되었다.

파이프라인의 단계별 구성은 다음과 같다.

| 단계 | 명칭 | 기능 | 처리 주체 |
|------|------|------|-----------|
| ① | Static Profile 매칭 | 앱 진입 시 고정된 사용자 성향(`target_orientation`)으로 1차 하드 필터링 | ChromaDB `where` 절 |
| ② | Query Intent Parsing | 사용자 발화에서 동적 장르·긍정/부정 카테고리 의도를 구조화 추출 | Claude Haiku 4.5 |
| ③ | Negative Screening | 기피 카테고리가 포함된 후보를 애플리케이션 레이어에서 배제 | Python 루프 |
| ④ | 3-Axis Reranking | 정규화 유사도·최신성·카테고리 매칭의 가중 합산 재정렬 | Python |
| ⑤ | Generation | 최종 1위 캐릭터에 빙의한 1인칭 추천 멘트 생성 | Claude Haiku 4.5 |

이 구조의 핵심적 강점은 **상이한 성격의 두 제약 조건을 계층적으로 분리**했다는 점에 있다. 사용자의 성향(남성향/여성향)은 세션 전체에 걸쳐 불변하는 정적 속성이므로, 매 질의마다 재추론할 필요 없이 진입 시점에 한 번 고정하여 ChromaDB의 메타데이터 필터로 위임한다. 이는 인덱스 레벨에서 탐색 공간 자체를 축소시켜 후속 연산의 부하를 경감한다. 반면 사용자가 특정 순간에 원하는 장르나 소재는 발화마다 가변적인 동적 속성이므로, LLM 기반 의도 파싱으로 매 질의 시점에 실시간 추론한다.

또한 본 아키텍처는 **검색 정합성과 의미적 유연성을 의도적으로 분담**시켰다. 기피 소재의 배제(③ Negative Screening)는 "단 하나라도 포함되면 탈락"이라는 결정론적(deterministic) 하드 규칙으로 처리하여 사용자의 명시적 거부 의사를 엄격히 보장하는 한편, 후보 간의 우열 판정(④ Reranking)은 연속적 점수 공간에서의 가중 합산이라는 소프트(soft) 방식으로 처리한다. 이로써 "절대 보고 싶지 않은 것"과 "상대적으로 더 보고 싶은 것"을 서로 다른 논리로 다루는 정교한 통제가 가능해진다.

특히 ③단계는 ChromaDB가 메타데이터 문자열에 대한 부분 일치(`LIKE`) 필터를 지원하지 않는다는 제약을 우회하기 위한 설계다. `category` 필드가 `"romance,daily_modern"`과 같은 쉼표 결합 다중값 문자열로 저장되는 구조상, 벡터 DB 레벨의 필터만으로는 부분 일치 배제가 불가능하다. 이를 애플리케이션 레이어의 집합 연산(`set` 교집합)으로 이관함으로써, 벡터 검색 엔진의 기능적 한계를 코드 레벨에서 완전하게 보완하였다.

## 2. 50개 뤼튼 크랙 미러링 데이터셋 설계 의도

v1 프로토타입의 검증은 실제 서비스 데이터가 아닌 **50개 규모의 합성 데이터셋(Synthetic Dataset)** 을 기반으로 수행되었다. 이는 개발 일정상의 편의가 아니라, 다음 두 가지 공학적 판단에 근거한 의도적 설계 결정이다.

**첫째, 데이터 수집 단계와 알고리즘 검증 단계의 관심사 분리(Separation of Concerns)다.** 실서비스 크롤링은 대상 플랫폼의 이용약관(ToS) 준수 여부에 대한 별도 검토를 요구하는 독립적 과제다. 추천 알고리즘의 변별력 검증이라는 본 단계의 목표는 데이터의 *출처*가 아니라 데이터의 *구조와 다양성*에 의존하므로, 크롤링 모듈을 잠정 유보하고 동등한 입력 규격을 갖춘 합성 데이터로 대체하는 것이 합리적이다. 시스템 입장에서 입력의 스키마와 분포가 동일하다면, 데이터의 출처는 알고리즘 검증의 타당성에 영향을 주지 않는다.

**둘째, 초기 모킹 데이터(3건)의 표본 부족 문제 해결이다.** 개발 초기 단계에서 사용된 3건 규모의 데이터로는 후보군 내 Min-Max 정규화의 기준점(최댓값·최솟값)이 극도로 불안정하며, 재정렬 알고리즘이 유의미한 변별을 수행할 수 없다. 50개 규모의 데이터 풀은 후보군의 통계적 안정성을 확보하고, 정서 스펙트럼 전반에 걸친 변별력 테스트를 가능하게 한다.

다만 본 데이터셋은 단순한 허구가 아니라 **실제 뤼튼 크랙 서비스 화면의 정밀 미러링(Mirroring)** 을 원칙으로 구축되었다. 1:1 롤플레잉 및 시뮬레이션 장르의 남성향·여성향 4개 조합에 대해 실제 캐릭터 상세 화면을 분석하여, 다음 요소를 현실에 부합하도록 반영하였다.

- **스키마 정합성**: 실제 UI에서 확인된 필드(제작자 닉네임, 장르 칩, 성향 칩, 카테고리 칩, 대화수·좋아요·댓글 메트릭, 업데이트 버전·일자, 자유 태그)를 분석하여 12개 필드의 메타데이터 스키마(v2)를 확정하였다.
- **메트릭의 멱법칙(Power-Law) 분포**: 소수의 네임드 캐릭터(대화수 수백만, 제작자 팔로워 수만)와 다수의 롱테일 캐릭터(대화수 수천~수만, 팔로워 수백)가 공존하는 실제 UGC 플랫폼의 비대칭 분포를 모사하였다.
- **태그의 무한성 반영**: 제작자가 자유 입력하는 태그(`tags_raw`)의 개수 편차(3~10개)와 비정형성을 보존하되, 정서/소재 상위 카테고리로 정규화한 `tags_normalized`를 별도 필드로 분리하였다.
- **최신성 분포의 의도적 설계**: 재정렬 알고리즘의 최신성 축이 변별력을 갖도록, 업데이트 일자를 약 2년 범위에 걸쳐 분산시키고 기준일(2026-05-19) 30일 이내의 신작을 의도적으로 다수 배치하였다.

결과적으로 본 데이터셋은 "출처는 합성이되 구조와 분포는 실데이터"라는 성격을 가지며, 향후 정식 크롤러 또는 공식 API가 연동될 경우 데이터 적재 계층(ChromaDB 메타데이터)과 추천 파이프라인을 수정 없이 그대로 재사용할 수 있도록 설계되었다.

## 3. 실제 테스트 시연 결과 분석

v1 파이프라인의 전 단계 동작을 검증하기 위해, 긍정 의도와 부정 기피 의사가 동시에 포함된 복합 질의를 입력하여 원스톱 테스트를 수행하였다. 테스트 조건과 결과는 다음과 같다.

**입력 조건**
- 고정 사용자 성향(Static Profile): `male`
- 사용자 발화: *"요즘 마감 때문에 밤새서 피곤해. 이세계 전생해서 모험하는 시뮬레이션 같은 거 없나? 아, 연애 위주 취향인 로맨스는 절대 싫어"*

**② Query Intent Parsing 결과**

Claude Haiku 4.5 기반 의도 파서는 위 비정형 발화로부터 다음의 구조화 JSON을 정확히 추출하였다.

```json
{
  "genre": "simulation",
  "categories_positive": ["sf_fantasy"],
  "categories_negative": ["romance"]
}
```

발화에 포함된 "시뮬레이션"이라는 장르 단서, "이세계 전생"이라는 SF/판타지 소재 단서, 그리고 "로맨스는 절대 싫어"라는 명시적 거부 의사가 각각 `genre`, `categories_positive`, `categories_negative` 필드로 정확히 매핑되었다. 이로써 수동 카테고리 선택 UI 없이 자연어만으로 다차원 필터 조건을 생성하는 인텐트 파싱 아키텍처의 실효성이 입증되었다.

**①③ 하드 필터링 및 네거티브 스크리닝 결과**

정적 성향 필터(`male` 및 공용 `unspecified` 허용)와 동적 장르 필터(`simulation`)를 적용하고, 기피 카테고리(`romance`)가 포함된 후보를 배제한 결과, 최종 13개의 후보가 재정렬 단계로 진입하였다. 후술하는 점수표에서 확인되듯 13개 후보는 **전원이 시뮬레이션 장르이며 어느 항목도 `romance` 카테고리를 포함하지 않아**, 하드 필터링과 네거티브 스크리닝이 명세대로 정확히 작동하였음이 검증되었다.

최종적으로 `심해 연구소 : 마지막 교신`이 0.9050점으로 1위에 선정되었으며, ⑤단계에서 해당 세계관에 빙의한 1인칭 추천 멘트가 정상 생성되었다.

**시연 결과의 종합 평가**

본 시연을 통해 v1 파이프라인의 5개 단계 — 의도 파싱, 정적 하드 필터링, 동적 네거티브 스크리닝, 3축 정규화 재정렬, 멘트 생성 — 가 **단절 없이 연동되어 기술적으로 완결성 있게 구동됨**이 검증되었다. 특히 raw유사도가 0.1690~0.3889라는 폭 0.22의 좁은 밴드에 압축되어 있음에도, Min-Max 정규화를 통해 0.0000~1.0000의 전 구간으로 재분포시킴으로써, 재정렬 알고리즘이 의미 있는 점수 변별을 수행할 수 있게 한 점은 의도된 설계의 정상 작동을 보여준다.

다만 후술하는 바와 같이, **파이프라인의 기술적 완결성과 추천 결과의 의미적 적합성(Relevance)은 별개의 문제**이며, 본 시연 결과는 동시에 v1의 구조적 한계를 드러내는 진단 자료이기도 하다.

## 4. 공학적 관점에서의 핵심 결함 및 한계점 진단

v1 프로토타입은 파이프라인 차원에서 정상 구동되었으나, 시연 로그에 대한 정밀 분석 결과 추천 품질을 저해하는 세 가지 구조적 결함이 식별되었다. 본 절은 이를 정량적 근거와 함께 고백적으로 진단한다.

### (A) Min-Max 정규화에 의한 미세 노이즈의 강제 만점화

3축 재정렬에서 유사도 축의 정규화는 필수 처치였다. OpenAI 임베딩의 raw 코사인 유사도는 후보 간 격차가 극도로 작아(본 시연에서 1위와 13위의 차이는 0.2199), 정규화 없이 가중 합산할 경우 0~1 범위로 분포하는 최신성·카테고리 축에 의해 유사도 축의 영향력이 완전히 잠식되는 스케일 불일치(Scale Imbalance) 현상이 발생한다. Min-Max 정규화는 이 문제를 해소하기 위한 정당한 설계 판단이었다.

그러나 이 처치는 **입력 신호 자체의 분산이 본질적으로 작다는 사실과 결합될 때 부작용을 낳는다.** Min-Max 정규화는 후보군의 최댓값을 무조건 1.0, 최솟값을 무조건 0.0으로 사상(寫像)하므로, **후보 간 절대적 격차의 크기 정보를 소실시킨다.** 본 시연에서 1위 `심해 연구소`의 raw유사도(0.3889)와 2위 `포스트 아포칼립스`의 raw유사도(0.3474)는 불과 0.0415 차이에 불과하나, 정규화 후에는 1.0000 대 0.8113으로 그 격차가 인위적으로 확대되었다. 즉 임베딩 모델이 사실상 "유의미한 차이를 판별하지 못한" 미세 노이즈 수준의 유사도 차이가, 정규화를 거치며 마치 명확한 우열이 존재하는 것처럼 **착시적으로 증폭**된 것이다.

이는 Min-Max 기법 자체의 결함이 아니라, *분산이 빈약한 입력에 순위 정규화를 적용한 조합*에서 비롯되는 구조적 한계다. 후보군의 표본 수가 작을 경우 단일 이상치가 최댓값·최솟값을 좌우하여 정규화 결과 전체가 출렁이는 불안정성 또한 동일한 원인에서 파생된다.

### (B) 카테고리 화이트리스트 압축에 의한 서사 정보 손실

v1은 LLM 의도 파서의 출력 안정성을 확보하기 위해, `categories_positive`/`categories_negative`의 값을 사전 정의된 10개 카테고리(`romance`, `bl`, `gl`, `sf_fantasy`, `romance_fantasy`, `martial_arts`, `daily_modern`, `period`, `adult`, `etc`) 화이트리스트로 강제 제약하였다. 이는 파서가 데이터셋에 존재하지 않는 임의의 단어를 생성하여 카테고리 매칭이 무력화되는 사태를 방지하기 위한 방어적 설계였다.

그러나 이 압축 과정에서 **사용자 발화에 담긴 핵심 서사 정보가 소실**되었다. 본 시연의 사용자 발화 "이세계 **전생**해서 **모험**하는"에서 '전생(轉生)'과 '모험'은 사용자가 원하는 콘텐츠의 서사적 핵심을 규정하는 키워드다. 그러나 이 두 개념은 10개 카테고리 화이트리스트에 대응 항목이 존재하지 않아, 파싱 결과에서 단지 `sf_fantasy`라는 광범위한 단일 상위 카테고리로 환원(reduction)되어 버렸다.

그 결과는 추천 품질에 직접적 손상으로 나타났다. 1위로 선정된 `심해 연구소 : 마지막 교신`은 `sf_fantasy` 카테고리에는 부합하나, '이세계 전생'도 '모험'도 아닌 심해 고립 스릴러다. 반대로 사용자의 발화 의도에 정확히 부합하는 `내 게임 속으로 빙의했다`(빙의·회귀·게임판타지)와 `이 선협식 헌터물은 난이도가 미쳤다`(전생·각성)는 각각 5위와 8위로 강등되었다. 카테고리 추상화의 입도(granularity)가 사용자 의도의 입도보다 거칠어, **세분화된 서사적 요구가 시스템에 전달되지 못한 정보 손실(Information Loss)** 이 발생한 것이다.

### (C) 단일 긍정 카테고리 입력 시 카테고리 축의 상수화

(B)의 정보 손실은 (C)의 변별력 무력화로 연쇄된다. 본 시연에서 `categories_positive`는 `sf_fantasy` 단 하나로 환원되었다. 카테고리 매칭 점수는 "캐릭터 카테고리 ∩ 사용자 긍정 카테고리"의 교집합 비율로 산출되므로, 긍정 카테고리가 단 하나일 경우 그 점수는 사실상 **"해당 카테고리 보유 여부"의 이진 지표**로 퇴화한다.

시연 점수표가 이를 명확히 입증한다. 13개 후보 중 10개가 카테고리 점수 1.0000을 획득하였고, 0.0000은 3개에 불과하다. 시뮬레이션 장르 후보의 대다수가 `sf_fantasy`를 보유하고 있어, 카테고리 점수가 후보군 전반에 걸쳐 1.0000으로 포화된 것이다. **후보 다수가 동일 점수를 가질 때 해당 축은 순위 변별에 기여하지 못한다.** 카테고리 축에 0.25의 가중치를 부여하였음에도, 실질적으로는 다수 후보에게 +0.25를 균일 가산하는 상수항(constant term)으로 작동하였을 뿐, 후보 간 우열을 가르는 변별 변수로서의 기능을 상실하였다.

결과적으로 v1의 최종 순위는 명목상 3축 가중 합산임에도, 카테고리 축의 상수화로 인해 **실질적으로는 정규화 유사도와 최신성이라는 2개 축에 의해 결정**되었다. 그리고 (A)에서 진단한 바와 같이 정규화 유사도 축마저 미세 노이즈의 증폭에 기반하므로, 1위 선정의 신뢰성은 공학적으로 충분히 담보되지 못한 상태다.

### 결함 진단의 종합

세 결함은 독립적이지 않고 **인과적으로 연쇄**되어 있다. (B) 화이트리스트 압축이 서사 정보를 소실시키고 → (C) 단일 카테고리로의 환원이 카테고리 축을 상수화하며 → 그 결과 순위 결정이 (A) 노이즈가 증폭된 유사도 축에 과도하게 의존하게 된다. 이 연쇄가 "기술적으로는 완벽히 구동되나 의미적으로는 사용자 의도를 빗나간" v1의 역설을 설명한다. 본 진단은 v1을 실패로 규정하는 것이 아니라, 차세대 개선의 방향을 정량적 근거 위에서 확정하기 위한 자기 검증 절차다.

## 5. 차세대(v2) 프로토타입 개선 로드맵

후속 로드맵은 두 갈래로 구성된다. **(가) 매칭 품질 개선** 은 본 장 4절에서 진단된 결함의 인과 연쇄를 역순으로 차단하는 과제이며, **(나) 자동화 데이터 파이프라인(Scaffolding) 구축** 은 v1 범위에서 의도적으로 분리(0절 참조)했던 프로젝트 계획서의 또 다른 핵심 축을 실체화하는 과제다. 전자가 "v1이 세운 매칭 뼈대를 정교화"하는 작업이라면, 후자는 "그 뼈대에 데이터를 자동으로 공급하는 혈관을 부설"하는 작업에 해당한다.

### 5.1 (가) 매칭 품질 개선 로드맵

v1에서 진단된 결함의 인과 연쇄를 역순으로 차단하기 위해, 다음의 개선을 우선순위에 따라 추진한다.

**개선 1 — 임베딩 텍스트에 서사 설명(Description) 통합 [최우선].** 현재 임베딩 입력 텍스트는 캐릭터명과 태그만으로 구성되어 있어, '전생', '모험', '고립'과 같은 서사적 맥락 신호가 벡터에 충분히 반영되지 못한다. 캐릭터의 상세 설명문(Description)을 임베딩 입력에 통합함으로써, 결함 (A)의 근본 원인인 raw유사도의 분산 빈약 문제를 입력 신호 차원에서 완화한다. 이는 코드 수정 범위가 작으면서 기대 효과가 큰 개선으로, v2의 1순위 과제로 설정한다.

**개선 2 — 카테고리 매칭의 `tags_normalized` 기반 다각도 매칭으로의 진화.** 결함 (B)·(C)의 핵심 원인은 10개로 제한된 거친 카테고리 체계다. v2는 매칭 기준을 `category`에서 보다 세분화된 `tags_normalized`(위로·생존·성장·액션·집착 등 약 29종의 정서/소재 지표)로 전환한다. 매칭 대상 어휘의 입도를 사용자 의도의 입도에 근접시킴으로써, 단일 카테고리로의 환원에 따른 정보 손실과 상수화 현상을 동시에 해소한다. 이때 의도 파서 또한 `tags_normalized` 어휘 체계를 출력하도록 프롬프트를 재설계하며, 해당 어휘 체계를 공식 표준 사전으로 확정한다.

**개선 3 (검토 과제) — 한국어 특화 임베딩 모델로의 전환 검토.** 결함 (A)의 보다 근본적 원인은 사용되는 임베딩 모델의 한국어 정서·서사 뉘앙스 변별력 한계에 있다. `text-embedding-3-large` 또는 한국어 특화 임베딩 모델(BGE-M3 등)로의 전환을 통한 raw유사도 분산 개선을 v2의 검토 과제로 둔다. 다만 모델 전환은 비용 구조와 임베딩 차원 변경에 따른 벡터 스토어 재구축을 수반하므로, 개선 1·2의 효과를 정량 측정한 후 비용 대비 효과를 평가하여 도입 여부를 결정한다.

**개선 4 (후속 과제) — 정량적 평가 체계(RAGAS / LLM-as-a-Judge) 구축.** 위 개선들의 효과를 객관적으로 검증하기 위해, 추천 적합성(Relevance)·설명력(Explainability)·사실 충실성(Faithfulness)을 수치화하는 자동 평가 하니스를 구축한다. 이를 통해 v1 대비 v2의 추천 품질 개선 폭을 정량 지표로 제시하며, "측정에 기반한 개선" 사이클을 확립한다.

### 5.2 (나) 자동화 데이터 파이프라인(Scaffolding) 구축 로드맵

v1은 매칭 엔진의 아키텍처 골격 수립에 범위를 한정하였으므로(0절 참조), 프로젝트 계획서가 핵심 목표로 제시한 자동화 데이터 파이프라인은 아직 구현되지 않았다. 현 시점에서 데이터 적재는 일회성 배치 스크립트의 수동 호출에 의존하며, 이는 계획서 3.2의 평가지표("데이터 최신성: 1일 1회 이상")와 4.1의 요구사항("신규 캐릭터 생성 시 즉시 벡터 스토어에 반영되는 실시간 동기화 로직")을 충족하지 못하는 상태다. 후속 마일스톤은 이 간극을 다음 단계로 해소한다.

**개선 5 — 증분 동기화(Incremental Synchronization) 파이프라인 구축.** 전체 데이터를 매번 재적재하는 현행 방식 대신, 신규 또는 변경된 캐릭터만을 식별하여 해당 항목만 임베딩·`upsert`하는 증분 동기화 로직을 구현한다. 이 단계는 실서비스 크롤러의 연동 여부와 무관하게, 합성 데이터셋의 변경 감지만으로도 파이프라인의 자동 반영 구조 자체를 실증할 수 있다. 즉 데이터의 *출처*는 모킹 상태로 두더라도, *동기화 메커니즘*은 실제로 동작하는 형태로 구축함으로써 계획서의 "실시간 동기화" 요구를 구조적으로 충족한다.

**개선 6 — 스케줄링 기반 자동 실행 체계 도입.** 증분 동기화 파이프라인을 스케줄러(예: 운영체제 수준의 cron, 또는 Python 기반 스케줄링)에 연결하여, 1일 1회 이상 자동 실행되는 무인(無人) 갱신 체계를 구축한다. 이로써 계획서 3.2의 데이터 최신성 평가지표를 정량적으로 만족시킨다.

**개선 7 (선행 검토 과제) — 데이터 수집 계층의 연동.** 자동화 파이프라인의 최종 입구에 해당하는 실데이터 수집 모듈은, 대상 플랫폼의 이용약관 및 `robots.txt`에 대한 검토와 공식 API 제공 여부 확인을 선행 조건으로 한다. 이 검토 결과에 따라 합법적 수집 범위가 확정되면, 개선 5에서 구축한 증분 동기화 파이프라인의 데이터 공급원을 모킹 데이터셋에서 실데이터로 교체한다. 이때 v1에서 안정화한 메타데이터 스키마(v2)가 인터페이스로 작동하므로, 수집 계층의 교체가 매칭 엔진의 수정으로 파급되지 않는다.

### 5.3 로드맵 종합

v1은 프로젝트 계획서가 제시한 두 축 가운데 (가) 지능형 매칭 엔진의 골격을 수립하였으며, 본 로드맵은 그 골격의 품질을 정교화하는 과제(5.1)와 (나) 자동화 데이터 파이프라인을 실체화하는 과제(5.2)를 병행 제시한다. 이로써 계획서의 모든 핵심 요구사항이 마일스톤 단위의 추진 계획 위에 빠짐없이 배치되며, v1의 의도적 범위 한정이 미완의 결손이 아니라 단계적 개발 전략의 일부임이 드러난다.
