# 캡스톤 디자인 2차 프로토타입 최종 보고서

> 본 보고서는 1차 프로토타입(v1) 보고서 이후 단계적으로 추진된 매칭 엔진 고도화(v2), 자동화 데이터 파이프라인 구축(v3), 자동 평가 하니스 구축(v4)의 전체 성과를 종합하는 2차 프로토타입 보고서다. 각 단계는 v1 보고서에서 진단된 결함과 프로젝트 계획서의 핵심 요구사항에 대한 직접적 응답으로 설계되었으며, 본 보고서는 그 설계 의도·구현 결과·정량 평가·잔존 한계를 학술적 톤으로 통합 기술한다.

## 0. 본 보고서의 범위 및 v1 보고서와의 관계

v1 보고서는 매칭 엔진의 아키텍처 골격을 수립하고 50개 합성 데이터셋 기반의 시연을 완료한 시점을 마일스톤으로 정의하였다. 동시에 v1 보고서는 자체 시연 로그에 대한 정밀 분석을 통해 다음 세 가지 결함을 정량적으로 진단하였다.

- **결함 (A)**: Min-Max 정규화가 분산 빈약한 raw 유사도와 결합되어 미세 노이즈를 인위적으로 증폭시키는 현상.
- **결함 (B)**: 10개 카테고리 화이트리스트로의 압축 과정에서 발생하는 핵심 서사 정보의 손실(Information Loss).
- **결함 (C)**: 단일 긍정 카테고리 입력 시 카테고리 매칭 점수가 후보 다수에서 1.0으로 포화되어 변별력을 상실하는 상수화 현상.

또한 v1 보고서는 프로젝트 계획서가 정의한 최종 시스템을 **(가) 지능형 매칭 엔진**과 **(나) 자동화 데이터 파이프라인(Scaffolding)**의 두 축으로 분해하고, v1이 (가) 축에 범위를 집중하였음을 명시하였다. (나) 축, 그리고 계획서 3장이 명시한 평가 체계는 후속 마일스톤의 과제로 분리되었다.

본 2차 보고서는 그 후속 마일스톤의 결과물이다. v2는 결함 (B)·(C)에 대한 매칭 엔진 차원의 응답이며, v3는 (나) 축 자동화 파이프라인의 실체화, v4는 계획서 3장 평가 체계의 자체 구현에 해당한다. 결함 (A)는 v2에서 부분 완화되었으며, 잔존 영향과 향후 처치 방향은 본 보고서 5장에서 다룬다.

## 1. 1차 프로토타입(v1) 결함 극복 및 매칭 엔진 고도화 (v2)

### 1.1 시맨틱 노이즈 및 서사 손실의 양면 방어

v1 보고서의 결함 (B)는 "이세계 전생", "모험"과 같은 사용자 발화의 서사적 핵심 키워드가 10개 카테고리(`sf_fantasy` 등)로 환원되며 소실되는 정보 손실 현상이었다. 이를 입력 신호 차원에서 보강하기 위한 1차 처치 후보는 캐릭터의 상세 설명문(`description`)을 임베딩 입력에 결합하는 방안이었다.

그러나 이 방안은 실제 대상 플랫폼인 뤼튼 크랙의 데이터 분석 결과 기각되었다. 캡처 분석을 통해 확인된 상세 설명 영역의 콘텐츠 형태는 다음과 같이 일관되지 않았다.

- 순수 줄글 서술 (예: 캐릭터 `최보윤` — "이별에 지친 여사친, 그녀를 위로해주자")
- 메타 안내문 (예: 캐릭터 `교도소` — "[인원:6명] TMI: 시각을 넘기셔도 되고...")
- 웹툰 이미지 콘텐츠 (텍스트 크롤링 불가)
- 외부 사이트로의 링크 안내

이러한 형식적 비일관성은 곧 데이터 품질의 비결정성을 의미하며, "[인원:6명]"과 같은 비서사적 메타 텍스트가 임베딩에 혼입될 경우 시맨틱 검색 품질을 도리어 저해한다. 따라서 본 프로젝트는 **수집 가능성이 검증되지 않은 description을 임베딩 입력에서 과감히 배제**하고, 안정적으로 수집 가능한 9개 필드(이름, 카테고리, 태그, 메트릭 5종, 업데이트일)만을 활용하는 방향으로 선회하였다.

이는 정보량과 수집 안정성 사이의 의도적 트레이드오프이며, 그 대가는 본 보고서 3.4절에서 평가 결과를 통해 정량적으로 측정된다.

### 1.2 장르별 임베딩 텍스트 분기 설계

description 배제의 손실을 보완하기 위해, 본 프로젝트는 캐릭터의 `genre` 값에 따라 임베딩 입력 텍스트를 다르게 구성하는 분기 로직을 `vector_store.py`의 `_build_embedding_text` 메서드에 도입하였다. 설계 근거는 두 장르의 정보 구조가 본질적으로 상이하다는 관찰에서 출발한다.

- **시뮬레이션(simulation)**: 제목 자체가 세계관·서사의 요약 기능을 수행한다. (예: "포스트 아포칼립스 : 마지막 분대" → 장르·상황이 제목에 명시적으로 노출됨.) 따라서 임베딩 텍스트에서 제목을 핵심 신호로 활용한다.
- **1:1 롤플레잉(roleplay_1on1)**: 제목이 대개 캐릭터의 고유명사이므로 서사 정보가 빈약하다. (예: "차도윤" → 이름만으로는 성격을 알 수 없음.) 캐릭터의 성격은 창작자가 자유 입력한 태그에 거의 전적으로 의존하므로, 임베딩에서 이름의 비중을 낮추고 태그를 핵심 신호로 삼는다.

이 분기는 "수집 가능한 필드만 사용한다"는 제약 하에서 정보 활용도를 극대화하기 위한 설계이며, 단일 임베딩 전략으로는 두 장르의 정보 구조 차이를 적절히 반영할 수 없다는 인식에 기반한다.

### 1.3 태그 커버리지 비율 공식의 채택

v1 결함 (C)는 10개 카테고리 매칭 점수가 단일 긍정 입력에서 상수화되는 현상이었다. v2는 매칭 대상을 거친 10개 카테고리에서 세분화된 29개 정규화 태그(`tags_normalized`)로 전환함으로써 이 결함의 근본 원인을 제거하였다.

매칭 함수 자체의 수식적 설계 역시 정밀하게 검토되었다. 최초 후보였던 자카드 유사도(Jaccard Similarity)는 다음 수식으로 정의된다.

```
Jaccard(A, B) = |A ∩ B| / |A ∪ B|
```

본 시스템의 운용 맥락에서 자카드 유사도는 수학적 맹점을 갖는다. 사용자의 누적 선호 태그 집합(대화 턴이 거듭될수록 6~8개로 확장)과 캐릭터의 정규화 태그 집합(평균 약 4개)은 크기 비대칭이 본질적이다. 자카드는 합집합을 분모로 사용하므로, **캐릭터가 사용자의 선호를 완벽히 충족(완전 포함)하더라도 두 집합의 크기 차이만큼 점수가 부당하게 하락**한다. 이는 추천 시스템에서 측정하고자 하는 본질("이 캐릭터가 사용자 취향을 얼마나 커버하는가")과 부합하지 않는다.

따라서 본 프로젝트는 분모를 사용자 선호 태그 수로 고정한 **커버리지 비율(Coverage Ratio)** 을 채택하였다.

```
Coverage(C, P) = |C ∩ P| / |P|

  C = 캐릭터의 tags_normalized 집합
  P = 사용자 누적 긍정 태그 집합 (accumulated_positives)
```

이 공식은 발화 길이에 따른 분모 팽창의 영향을 받지 않으며, "캐릭터가 사용자가 원한 N개 태그 중 몇 개를 보유하는가"라는 직관적 의미를 직접 측정한다. v1 결함 (C)의 상수화 문제 역시 매칭 어휘가 10개에서 29개로 확장되며 구조적으로 해소되었다.

### 1.4 4축 가중 합산 랭킹 포뮬러

위 변경을 반영하여, 최종 재정렬 점수는 다음 4축 가중 합산으로 산출된다.

```
final_score = 0.35 × normalized_similarity     (유사도 축)
            + 0.30 × tag_coverage              (태그 매칭 축)
            + 0.20 × normalized_quality        (품질 축)
            + 0.15 × recency_score             (최신성 축)
```

여기서 품질 축은 인기 편향 억제를 위해 신규 도입된 축이며, 좋아요 비율에 베이지안 평균 보정을 적용한 값을 후보군 내 Min-Max 정규화한 결과다. 정규화 유사도와 정규화 품질 모두 후보군 내 분산이 작은 raw 신호를 [0, 1] 범위로 펼쳐 가중치가 실질적으로 작동하게 한다. v1 보고서에서 결함 (A)로 지적한 "정규화가 노이즈를 증폭시키는" 부작용은 vector_store 단계의 `threshold_guard` (절대 유사도 하한선)와 `min_survivors` 폴백을 통해 입력단에서 1차 차단된다.

계획서가 명시한 인기 편향 해소 과제에 대한 직접적 응답으로서, 본 시스템은 `creator_followers`와 `conversation_count`를 점수 계산에 일절 사용하지 않는다. 대신 규모와 독립적인 좋아요 *비율*을 품질의 대리 지표로 사용하여, 대화수가 적은 롱테일 신작도 만족도가 높으면 정당하게 평가받도록 설계되었다.

### 1.5 다턴 누적 프로파일링 (UserProfileManager) 도입

v1은 매 질문을 독립적으로 처리하는 단발성(stateless) 구조였다. v2는 `src/recommender/profile_manager.py`에 `UserProfileManager`를 신설하여, 대화 턴을 거치며 사용자 취향을 누적(stateful)하는 구조로 패러다임을 전환하였다.

`UserProfile`은 다음 4개 필드로 구성된다.

- `accumulated_positives`: 누적 긍정 선호 태그 (집합 누적)
- `accumulated_negatives`: 누적 기피 태그 (집합 누적, 동시에 모순 해소 규칙으로 긍정 목록에서 제거)
- `preferred_genre`: 선호 장르 (최신 발화로 덮어쓰기)
- `current_mood`: 현재 정서 (세션 내 누적, 다른 정서는 이어붙임)

이 구조는 사용자가 N번째 턴에서 "로맨스 빼줘"라고 발화하면, 이후의 모든 추천에서 해당 기피 조건이 영구 적용됨을 의미한다. 이는 계획서 1.3.3 "대화형 큐레이션 인터페이스"의 단발성 한계를 직접적으로 극복하는 구조다.

## 2. 윤리적·안전한 동적 크롤링 파이프라인 구축 (v3)

### 2.1 가상 미러링 사이트 도입의 정당성

본 프로젝트의 v3 단계는 계획서 1.3.2가 명시한 "완전 자동화 파이프라인"의 실체화를 목표로 한다. 그러나 자동화 파이프라인의 데이터 입구에 해당하는 실서비스 크롤링 모듈을 곧바로 구현하는 데에는 다음의 검토가 선행되어야 한다.

본 프로젝트는 기업 연계형 캡스톤으로서, 착수회의를 통해 뤼튼 크랙의 공개 AI 작품 데이터 수집에 대한 협의를 거쳤다. 그럼에도 불구하고 실서비스를 대상으로 한 즉각적인 자동 크롤링은 다음의 기술적·윤리적 리스크를 수반한다.

- **자동화 봇 차단 정책**: 일반적으로 현대 웹 서비스는 `robots.txt` 및 보안 인프라(Cloudflare 등)를 통해 자동화 트래픽을 통제하며, 학술 목적이라 하더라도 명시적으로 허용되지 않은 자동 수집은 약관 위반 소지가 존재한다.
- **운영 부담 전가**: 개발·디버깅 과정에서 반복되는 실서비스 호출은 상대 서버에 불필요한 부하를 야기한다.
- **시연 재현성 손상**: 발표 시연 도중 실서비스의 응답 지연 또는 UI 변경이 데모 실패로 직결될 수 있다.

이러한 검토에 따라 본 프로젝트는 v3 시연 단계에서 **실서비스 DOM 구조를 모사한 로컬 가상 미러링 HTML**(`data/mock_site/characters.html`)을 크롤링 타깃으로 채택하였다. 이는 데이터 수집의 *합법성* 문제가 아니라, *재현성·전환용이성·운영 윤리*를 종합 고려한 엔지니어링 결정이다.

가상 타깃의 DOM 구조는 캡처 분석을 통해 확인된 실서비스의 스토리 정보 레이아웃(제목, 제작자, 칩, 자유 태그, 메트릭, 업데이트일)을 정밀하게 모사한다. 이로써 v3 크롤러의 아키텍처(브라우저 구동, DOM 파싱, Rate Limit 가드)는 실전 그대로 구현되며, 향후 실서비스 전환 시 `target_url`과 CSS 셀렉터 문자열만 교체하면 수정 없이 동작한다.

본 보고서가 강조하는 바는, 가상 미러링이 "회피"가 아니라 "구조적 동등성을 유지한 채 외부 의존성을 절연하는" 적극적 설계 선택이라는 점이다. v3에서 검증하고자 하는 명제는 "뤼튼에서 데이터를 실제로 빼올 수 있는가"가 아니라, "동적 크롤링 → 정규화 → 증분 적재 → RAG 추천이라는 자동화 파이프라인이 구조적으로 완결되는가"이며, 이 명제는 가상 타깃으로 완전하게 검증된다.

### 2.2 Playwright 기반 동적 크롤러 및 Rate Limit 가드

v3 크롤러(`src/crawler/collector.py`의 `DynamicCharacterCrawler`)는 JavaScript 동적 렌더링 페이지를 처리하기 위해 Playwright 기반 헤드리스 Chromium을 구동한다. 정중한 크롤링(Polite Crawling) 원칙에 따라 다음의 가드를 장착한다.

- **무작위 지연(Random Jitter)**: 각 카드 파싱 사이에 `random.uniform(2.0, 5.0)` 초의 가변 대기를 주입한다. 기계적으로 일정한 간격의 요청은 상대 서버에 식별 가능한 부하 패턴을 만들고 차단 대상이 되므로, 사람과 유사한 불규칙 간격으로 요청한다.
- **커스텀 User-Agent**: `WrtnCapstoneResearchBot/1.0 (Academic capstone project; contact: ...)` 형식으로 학술 목적과 연락처를 명시한다. 단, 본 프로젝트는 이 User-Agent 자체가 수집 허가의 근거가 되지 않음을 명확히 인식한다. 정중한 신원 표시일 뿐, 실제 수집의 합법성은 대상 사이트의 약관과 robots.txt에 의해서만 정해진다.

이러한 가드는 가상 타깃 시연에서는 작동하지만 실효적 의미가 없으며, 향후 실서비스 전환 시점을 대비한 모범 사례 사전 장착에 해당한다.

### 2.3 데이터 정규화 파이프라인

크롤러가 수집한 날것의 데이터(이름, 칩 리스트, 태그 문자열, 축약 메트릭, 점 구분 날짜 등)는 v2 스키마 규격으로 정규화되어야 한다. 이 변환은 `vector_store.py`의 `_normalize_raw` 메서드에서 수행되며, 처리 항목은 다음과 같다.

- **칩 분해**: 단일 칩 리스트(`["시뮬레이션", "남성향", "SF/판타지"]`)를 사전 정의된 매핑 테이블을 통해 `genre`, `target_orientation`, `category` 세 필드로 분리한다.
- **날짜 형식 변환**: 화면 표기 형식(`2026.05.10`)을 본 시스템 표준 형식(`2026-05-10`)으로 변환하여 `_recency_score` 메서드의 `strptime` 파싱과 호환되게 한다.
- **자유 태그 정제**: `#던전 #생존` 형식의 텍스트를 `던전,생존` 형식으로 정규화하여 `tags_raw` 필드에 저장한다.
- **태그 표준화 (LLM 매핑)**: 무한 어휘의 자유 태그를 29개 표준 정서 태그 화이트리스트에 매핑하기 위해, 별도의 Claude Haiku 호출을 수행한다. 매핑 결과는 다시 화이트리스트로 교차 검증된 후 `tags_normalized` 필드에 저장된다.

마지막 단계의 LLM 호출은 캐릭터당 1회 발생하며, 현 규모(5~50개)에서는 비용·지연이 무시할 수준이다. 다만 향후 수천 건 규모로 확장될 경우 사전 매핑 우선 + LLM 폴백의 하이브리드 구조로 전환이 권장되며, 이는 본 보고서 5장에 후속 과제로 명기한다.

### 2.4 멱등성 보장 증분 적재

자동화 파이프라인은 동일 데이터의 반복 수집을 전제로 한다. 따라서 적재 단계는 같은 식별자의 데이터가 다시 입력되어도 중복 행을 생성하지 않는 멱등성(Idempotency)을 보장해야 한다. `upsert_crawled_characters` 메서드는 ChromaDB의 `upsert` 연산을 활용하여 이 속성을 구현한다. 동일 `character_id`가 이미 존재할 경우 메타데이터·임베딩이 갱신되며, 새 ID는 삽입된다.

이는 단순한 구현상의 디테일이 아니라, 계획서 4.1이 요구한 "신규 캐릭터 생성 시 즉시 벡터 스토어에 반영되는 실시간 동기화 로직"의 핵심 전제 조건이다. 같은 크롤링 스크립트가 매일 실행되어도 DB가 오염되지 않음이 보장되어야, 비로소 무인 자동 갱신 체계의 구축이 가능해진다.

### 2.5 실시간 RAG 대화 루프 (`chat_with_character`)

v2까지의 시스템은 1위 캐릭터에 대해 단발성 추천 멘트 1회를 생성하고 종료되었다. v3는 사용자가 선정된 캐릭터와 멀티턴 대화를 연속 진행하는 RAG 챗봇 루프로 진화하였다. 이는 계획서 1.3.3 "대화형 큐레이션 인터페이스"의 본격적 실체화에 해당한다.

`CharacterRecommenderEngine.chat_with_character` 메서드는 두 가지 컨텍스트 증강을 수행한다.

- **캐릭터 페르소나 주입**: 1위 캐릭터의 `name`, `genre`, `tags_raw`, `tags_normalized`를 system 프롬프트에 고정하여 캐릭터 정체성을 일관되게 유지한다. 응답 톤은 "2~3문장 간결, 목록 나열 금지"로 명시적으로 제약하여, 어시스턴트형 응답이 아닌 캐릭터다운 짧은 대사가 유도되도록 한다.
- **대화 기록 누적**: 이전 턴들의 user/assistant 메시지를 `messages` 리스트에 누적 주입한다. 이로써 Claude는 다음 응답 생성 시 이전 대화의 전체 맥락을 참조하며, 사용자가 후속 질문을 던질 때 그 맥락 위에서 연속성 있는 응답을 생성한다.

실제 시연 로그에서 1턴 캐릭터 응답이 "S급 마기, 입구 붕괴" 설정을 제시하면, 2턴의 사용자 후속 질문("살아남으려면 뭘 먼저?")에 대해 캐릭터가 "북쪽 능선을 타고, 음파 감지에 걸리지 말고"와 같이 1턴 설정을 정확히 이어받아 응답함이 확인되었다.

## 3. LLM-as-a-Judge 자동 평가 하니스 구축 및 결과 (v4)

### 3.1 RAGAS 라이브러리 비도입 결정

계획서 3.1은 평가 신뢰도 고도화 방안으로 RAGAS(Retrieval-Augmented Generation Assessment) 프레임워크 도입을 명시하였다. 그러나 v4 구현 단계에서 다음의 검토 결과 RAGAS 라이브러리의 직접 도입은 기각되었다.

- **의존성 충돌**: RAGAS는 LangChain 기반 의존성 체인을 요구하며, 이는 본 프로젝트가 일관되게 견지해온 "프레임워크 비도입, 순수 SDK" 아키텍처 원칙과 정면 충돌한다.
- **블랙박스화 우려**: 평가 로직이 외부 패키지에 위임될 경우, 채점 기준의 세부 조정·디버깅이 어려워지며 발표 단계에서의 설명 가능성이 저하된다.

이에 본 프로젝트는 **RAGAS의 평가 *철학*은 내재화하되 *라이브러리*는 사용하지 않는** 자체 구현 노선을 선택하였다. 구체적으로는 검색 품질(Faithfulness)과 생성 품질(Relevance, Explainability)을 분리하여 측정하는 RAGAS의 핵심 사상을 프롬프트 엔지니어링으로 직접 구현하였다.

### 3.2 교차 검증 아키텍처: 생성과 평가의 모델 분리

LLM-as-a-Judge 패러다임의 가장 흔한 함정은 동일 모델이 자기 생성물을 채점할 때 발생하는 자기편향(self-preference bias)이다. 본 시스템에서 추천 멘트의 생성자는 Claude Haiku이므로, 평가까지 Claude로 수행할 경우 객관성이 구조적으로 훼손된다.

이를 차단하기 위해 본 프로젝트는 **생성은 Claude, 평가는 OpenAI GPT가 수행하는 교차 검증 구도**를 채택하였다. 이는 계획서 3.1이 평가자 모델을 "GPT-4o 등"으로 예시한 의도와도 정확히 일치한다. 실제 구현에서 사용된 판정자 모델은 2026년 5월 기준 OpenAI의 저지연·저비용 모델인 **`gpt-5.4-mini`** 다. 채점이라는 작업의 특성상 플래그십 모델(gpt-5.5)의 추론 능력은 과잉이며, mini 계열이 JSON 출력 및 CoT 추론 능력을 충분히 제공한다.

### 3.3 Structured Outputs 기반 신뢰성 확보

판정자 응답의 JSON 구조 일관성은 평가 시스템의 신뢰성을 좌우하는 핵심 요소다. 구현 초기에는 `response_format={"type": "json_object"}` 옵션만을 사용하였으나, 이는 "유효한 JSON으로 답하라"는 지시일 뿐 구체적 구조를 강제하지 않는다. 그 결과 판정자가 호출마다 JSON 중첩 구조를 비결정적으로 변경하는 현상이 관측되었다. 식별된 변형 패턴은 다음과 같다.

- 정상: `{"reasoning": {...}, "scores": {...}}`
- 변형 1: `{"reasoning": {..., "scores": {...}}}` (scores가 reasoning 내부에 중첩)
- 변형 2: `{"reasoning": {"relevance": "4", ...}}` (reasoning 필드에 점수만 채움, scores 객체 자체가 부재)

코드의 점수 추출 로직(`parsed.get("scores", {}).get(key, 0)`)은 정상 구조만을 가정하였기에, 변형 패턴 발생 시 기본값 0이 점수로 둔갑하는 측정 실패가 빈발하였다. 더욱 심각한 것은 실패가 비결정적이어서 동일 시나리오가 매 실행마다 다른 결과를 산출한다는 점이었다.

본 프로젝트는 이를 임시방편(`get` 체인의 다중 경로 탐색)으로 해결하지 않고, OpenAI Structured Outputs의 `json_schema` 기능을 통해 **출력 구조를 스키마 차원에서 강제**하는 근본 해결을 채택하였다. 정의된 스키마(`EVALUATION_SCHEMA`)는 다음을 강제한다.

- 최상위에 `reasoning`과 `scores`가 형제 객체로 반드시 존재.
- 각 객체 내부에 `relevance`, `explainability`, `faithfulness` 3개 키가 정확히 존재 (`additionalProperties: false`).
- 각 점수는 정수형으로 강제.

`strict: True` 옵션 하에서 OpenAI는 모델 출력을 이 스키마에 강제로 부합시키며, 따라서 키 누락·중첩 변경이 구조적으로 불가능하다. 적용 이후 7개 시나리오 전체가 동일 구조의 응답으로 안정 측정되었다.

### 3.4 CoT 기반 채점 절차 및 평가 결과

판정자 system 프롬프트는 채점 순서를 다음과 같이 명시적으로 제약한다.

1. 각 지표(적합성·설명력·충실성)에 대한 채점 근거(reasoning)를 먼저 자연어로 서술.
2. 그 reasoning을 근거로 1~5점 정수 점수를 산출.

이 순서 강제는 LLM 판정자가 점수를 먼저 결정하고 사후적으로 정당화하는 휴리스틱적 채점을 방지하기 위한 설계다.

7개 시나리오 데이터셋(순수 감성형 2종, 장르+기피 조건형 2종, 서사 기반 액션/생존형 2종, 모호한 입력형 1종)에 대한 실제 평가 결과는 다음과 같다.

| 시나리오 | 유형 | 추천된 캐릭터 | 적합성 | 설명력 | 충실성 |
|---|---|---|---|---|---|
| A-1 | 순수 감성/위로형 | 한밤의 상담실 | 5 | 4 | 5 |
| B-1 | 장르+하드 기피 조건형 | 여름방학, 시골 할머니 댁 | 3 | 3 | 4 |
| C-1 | 서사 기반 액션/생존형 | 이 선협식 헌터물은 난이도가 미쳤다 | 4 | 3 | 4 |
| D-1 | 모호한 입력 | 옆자리 빌런이 자꾸 말 검 | 4 | 4 | 4 |
| A-2 | 순수 감성/위로형 (변형) | 한밤의 상담실 | 4 | 3 | 4 |
| B-2 | 장르+기피 조건형 (변형) | 무심한 천재 피아니스트 | 2 | 2 | 3 |
| C-2 | 서사 기반 액션/생존형 (변형) | 황야의 현상금 사냥꾼 | 2 | 2 | 4 |
| **평균** | | | **3.43** | **3.00** | **4.00** |

종합 평균은 **3.48 / 5.00**이다. 각 지표의 의미는 다음과 같이 해석된다.

- **충실성 4.00**: 멘트 생성기가 캐릭터 프로필 외 정보를 환각(hallucinate)하지 않고 주어진 태그·장르 범위 내에서 응답을 구성하고 있음을 시사한다. 본 시스템의 RAG 구조가 생성 단계에서 견고하게 작동함을 의미하는 가장 높은 점수다.
- **적합성 3.43**: 유사도·태그 매칭·품질·최신성을 결합한 4축 랭킹이 다수의 시나리오에서 사용자 의도에 부합하는 캐릭터를 1위로 선정함을 보인다. 단, 특정 유형(B-2, C-2)에서 점수가 무너지는 현상이 식별되며 이는 본 보고서 5장에서 별도 진단한다.
- **설명력 3.00**: 세 지표 중 일관되게 가장 낮으며, 그 원인은 본 보고서 5.1절에서 구조적으로 분석한다.

## 4. 시스템 아키텍처 종합도

본 보고서에 기술된 v2~v4의 모든 컴포넌트는 v1에서 수립된 폴더 구조 위에 누적적으로 통합되었다. 최종 시스템의 모듈 구성과 데이터 흐름은 다음과 같다.

```
[크롤러] (v3, src/crawler)
    ↓ raw 데이터
[정규화 파이프라인] (v3, src/store)
    ↓ 스키마 v2 규격 데이터
[벡터 저장소 ChromaDB] (v2/v3, src/store)
    ↑ ↓ 임베딩 · 메타데이터 (멱등 upsert)
[추천 엔진] (v2, src/recommender/engine)
    ← 누적 유저 프로필 (v2, src/recommender/profile_manager)
    ↓ 1위 캐릭터 + 추천 멘트
[RAG 대화 루프] (v3, src/recommender/engine.chat_with_character)
    ↓ 추천 결과
[평가 하니스] (v4, src/evaluation/harness)
    ← OpenAI GPT 판정자 (교차 검증)
```

각 단계는 인터페이스(메타데이터 스키마, 메서드 시그니처)를 통해 결합되며, 단일 컴포넌트의 교체가 다른 컴포넌트의 수정을 요구하지 않는다. 특히 가상 미러링 타깃을 실서비스 URL로 교체하는 단일 변경만으로 전체 자동화 파이프라인이 실데이터에 적용 가능하도록 설계되었다.

## 5. 핵심 결함 진단 및 후속 개선 방향

v4 평가 결과는 v2/v3 시스템의 두 가지 구조적 한계를 정량적으로 노출하였다. 본 절은 이를 회피하지 않고 정면으로 진단한다.

### 5.1 설명력 지표의 구조적 저하 — description 배제의 트레이드오프

7개 시나리오 전체에서 설명력 평균이 3.00으로 적합성(3.43)·충실성(4.00)에 비해 일관되게 낮다. 판정자가 제공한 reasoning을 분석하면 이 현상의 원인은 일관된 패턴을 보인다.

> "캐릭터의 태그를 직접 짚어 설명하기보다 분위기 묘사 위주이고, 왜 이 캐릭터인지에 대한 구조적 설명력은 보통 수준" (B-1 평가)
> "추천 사유를 정리해 제시하기보다 캐릭터 콘셉트를 대사처럼 풀어놓는 형태라, 납득 가능성은 있지만 설명 구조는 다소 약함" (D-1 평가)

이 패턴은 본 보고서 1.1절에서 기술한 **description 필드 배제의 직접적 귀결**이다. 멘트 생성기가 활용할 수 있는 캐릭터 정보가 이름·태그·장르·카테고리로 한정되어 있어, 캐릭터의 서사적 배경에 기반한 깊이 있는 추천 사유를 구성할 재료가 부족하다. 적합성과 충실성은 태그 정보만으로도 일정 수준 확보 가능하지만, 설명력은 서사적 맥락이 있어야 비로소 향상되는 지표다.

이는 v2 설계 단계에서 인지되고 의도적으로 수용된 트레이드오프다. description 수집의 비결정성을 회피한 대가로 설명력 지표에서 약 1점의 손실이 발생함이 본 v4 평가를 통해 정량 측정되었으며, 이 측정 자체가 v5에서의 처치 우선순위 결정에 기여한다.

### 5.2 멀티턴 변형 시나리오에서의 적합성 붕괴 — 긍정 의도 매칭의 약점

B-2와 C-2 시나리오에서 적합성이 2점으로 다른 시나리오의 4~5점에 비해 현저히 낮다. 두 시나리오의 사용자 의도와 시스템 추천을 비교한다.

- **B-2**: 사용자는 "어두운 거 빼줘, 밝은 게 좋아"를 명시. 그러나 추천된 캐릭터는 `무심한 천재 피아니스트`로 무심함·과묵함 중심이며, 판정자는 "유저의 '밝은 게 좋아'를 적극 반영했다고 보기 어렵다"고 평가.
- **C-2**: 사용자는 "치열한 전투와 전략이 살아있는 무협 세계관"을 명시. 그러나 추천은 `황야의 현상금 사냥꾼`(서부극)으로, 판정자는 "세계관과 정서가 크게 다르다"고 평가.

이 두 사례에서 공통적으로 관찰되는 패턴은 다음과 같다. **시스템의 네거티브 스크리닝(기피 조건 배제)은 정상 작동하나, 사용자가 적극적으로 표현한 긍정 의도("밝은", "무협")가 추천 결과에 충분히 반영되지 않는다.** 즉 본 시스템은 "싫은 것을 거르는 능력"은 갖추었으나 "원하는 것을 적극 매칭하는 능력"이 상대적으로 약하다.

원인은 두 갈래로 추정된다. 첫째, 한국어 임베딩 모델이 "밝은"·"무협"과 같은 정서·장르 키워드의 미묘한 함의를 충분히 변별하지 못하는 한계. 둘째, 4축 랭킹에서 긍정 의도가 주로 태그 커버리지 축(가중치 0.30)에서 평가되나, 사용자의 자연어 발화에서 추출된 태그가 표준 어휘로 환원되며 의미가 일부 손실되는 현상.

이는 새롭게 발견된 결함이며, v5 단계에서 다음의 처치가 검토된다.

- 긍정 의도 강도에 따른 동적 가중치 조정 (강하게 표현된 긍정 키워드에 가중치 부스트).
- 한국어 특화 임베딩 모델로의 전환 (예: BGE-M3 등).
- 의도 추출 단계에서 표준 어휘 매핑 외에 원본 키워드 보존 채널 추가.

### 5.3 잔존 결함 (A) — 정규화의 절대 격차 소실

v1 보고서에서 진단된 결함 (A) — Min-Max 정규화가 후보군 내 절대 격차를 보존하지 않는 성질 — 은 v2에서 입력단의 `threshold_guard`로 1차 차단되었으나 본질적으로 해소되지는 않았다. 평가 결과에서도 raw 유사도 차이가 미미한 후보들 사이에서 정규화 후 격차가 인위적으로 확대되는 현상이 관찰된다. 다만 4축 랭킹 구조 하에서 유사도 단일 축의 비중이 0.35로 축소되었고, 태그 매칭·품질·최신성이 보조 변별을 제공하므로, 결함 (A)의 실질적 영향은 v1 대비 상당히 완화되었다.

## 6. 결론

본 2차 보고서는 1차 프로토타입(v1)에서 진단된 결함을 매칭 엔진 차원에서 극복(v2)하고, 계획서가 요구한 자동화 파이프라인을 실체화(v3)하였으며, 평가 체계를 자체 구현하여 시스템 품질을 정량 측정(v4)한 결과를 종합한다. 종합 평균 3.48 / 5.00이라는 평가 점수는 본 시스템이 의미 있는 추천 품질에 도달했음을 보이는 동시에, 설명력 저하와 긍정 의도 매칭 약점이라는 잔존 과제를 정량 노출한다.

본 프로젝트가 견지한 일관된 원칙은 다음과 같다. 첫째, 결함을 은폐하지 않고 정량적 근거 위에 진단한다. 둘째, 외부 프레임워크에 의존하지 않고 순수 SDK로 모든 핵심 로직을 투명하게 구현한다. 셋째, 데이터 수집의 윤리적·기술적 리스크를 회피하기 위해 구조적 동등성을 유지한 가상 시연 환경을 구축한다. 넷째, 모든 설계 결정은 트레이드오프를 명시적으로 인식한 위에서 수행된다.

향후 v5에서는 본 보고서 5장에 명기된 결함 — 특히 5.2절의 긍정 의도 매칭 약점 — 을 우선 처치 대상으로 삼아, 본 v4 평가 점수의 정량적 개선을 추구한다. 그 개선 폭은 본 v4 하니스를 통해 객관적으로 측정될 것이며, 이는 "측정에 기반한 개선" 사이클의 두 번째 회전에 해당한다.
